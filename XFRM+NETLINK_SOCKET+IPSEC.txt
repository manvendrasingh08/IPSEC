------------------------------------------------------------------------------
1. WHAT IS XFRM?
------------------------------------------------------------------------------

XFRM = Linux kernel transformation framework used for:
- IPsec
- packet encryption/decryption
- authentication
- Security Associations (SA)
- Security Policies

It lives INSIDE the Linux kernel networking stack.

Main work:
- Apply ESP/AH transforms
- Encrypt outbound packets
- Decrypt inbound packets
- Match policies

------------------------------------------------------------------------------
2. WHAT ARE NETLINK SOCKETS?
------------------------------------------------------------------------------

Netlink sockets provide communication between:

    USER SPACE  <--------->  KERNEL SPACE

Used for:
- routing
- interfaces
- firewall
- networking configuration
- XFRM/IPsec configuration

------------------------------------------------------------------------------
3. RELATION BETWEEN XFRM AND NETLINK
------------------------------------------------------------------------------

User-space tools configure XFRM using Netlink sockets.

Examples:
- ip xfrm
- strongSwan
- libreswan

Flow:

    User Tool
        |
        |  Netlink Socket
        v
    NETLINK_XFRM
        |
        v
    Linux Kernel XFRM subsystem
        |
        v
    SPD + SAD updated

6. IMPORTANT XFRM STRUCTURES
------------------------------------------------------------------------------

1. Security Association info:

    struct xfrm_usersa_info

--------------------------------------------------

2. Policy info:

    struct xfrm_userpolicy_info

- action

------------------------------------------------------------------------------
7. IMPORTANT XFRM MESSAGE TYPES
------------------------------------------------------------------------------

XFRM_MSG_NEWSA        -> Add SA
XFRM_MSG_DELSA        -> Delete SA
XFRM_MSG_NEWPOLICY    -> Add policy
XFRM_MSG_DELPOLICY    -> Delete policy

- Type

------------------------------------------------------------------------------
11. SPD vs SAD
------------------------------------------------------------------------------

SPD = Security Policy Database
--------------------------------
Defines:
- WHICH traffic needs protection
- WHAT action to take

Example:
Traffic from:
10.0.0.1 -> 10.0.0.2

Use:
ESP transport mode

--------------------------------------------------

SAD = Security Association Database
-----------------------------------

Contains actual crypto details:
- SPI
- encryption keys
- auth keys
- algorithms
- tunnel/transport mode

14. HANDS-ON LAB USING NETWORK NAMESPACES
------------------------------------------------------------------------------

STEP 1 -> Create namespaces

sudo ip netns add ns1
sudo ip netns add ns2

--------------------------------------------------

STEP 2 -> Create virtual ethernet pair

sudo ip link add veth1 type veth peer name veth2

--------------------------------------------------

STEP 3 -> Move interfaces into namespaces

sudo ip link set veth1 netns ns1
sudo ip link set veth2 netns ns2

--------------------------------------------------

STEP 4 -> Configure IP addresses

sudo ip netns exec ns1 ip addr add 10.0.0.1/24 dev veth1

sudo ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth2

--------------------------------------------------

STEP 5 -> Bring interfaces up

sudo ip netns exec ns1 ip link set veth1 up
sudo ip netns exec ns1 ip link set lo up

sudo ip netns exec ns2 ip link set veth2 up
sudo ip netns exec ns2 ip link set lo up

--------------------------------------------------

STEP 6 -> Test connectivity

sudo ip netns exec ns1 ping 10.0.0.2

------------------------------------------------------------------------------
15. ADD XFRM SECURITY ASSOCIATIONS
------------------------------------------------------------------------------

Namespace 1:

sudo ip netns exec ns1 ip xfrm state add \
src 10.0.0.1 dst 10.0.0.2 \
proto esp spi 0x100 \
mode transport \
auth sha256 0x11111111111111111111111111111111 \
enc aes 0x22222222222222222222222222222222

--------------------------------------------------

Namespace 2:

sudo ip netns exec ns2 ip xfrm state add \
src 10.0.0.2 dst 10.0.0.1 \
proto esp spi 0x101 \
mode transport \
auth sha256 0x11111111111111111111111111111111 \
enc aes 0x22222222222222222222222222222222

------------------------------------------------------------------------------
16. ADD XFRM POLICIES
------------------------------------------------------------------------------

Namespace 1:

sudo ip netns exec ns1 ip xfrm policy add \
src 10.0.0.1/32 dst 10.0.0.2/32 \
dir out tmpl src 10.0.0.1 dst 10.0.0.2 \
proto esp mode transport

--------------------------------------------------

Namespace 2:

sudo ip netns exec ns2 ip xfrm policy add \
src 10.0.0.2/32 dst 10.0.0.1/32 \
dir out tmpl src 10.0.0.2 dst 10.0.0.1 \
proto esp mode transport

------------------------------------------------------------------------------
17. OBSERVE ENCRYPTED TRAFFIC
------------------------------------------------------------------------------

Run tcpdump:

sudo ip netns exec ns1 tcpdump -i veth1

Now ping again.

You should see:

ESP(spi=0x100,...)

instead of plain ICMP.

- better scalability
- better networking integration
